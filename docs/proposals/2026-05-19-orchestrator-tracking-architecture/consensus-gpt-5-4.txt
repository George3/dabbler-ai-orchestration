Findings

High: The Q6 / ship-plan section treats the multi-in-progress banner like a presentation choice, but in the current implementation it is the visible symptom of a fail-closed resolver. In proposal.md through proposal.md, the text implies 0.17.x could remove the banner while keeping the underlying writer model. That does not line up with the current reader contract in MarkerWatchService.ts and MarkerWatchService.ts, which returns unresolved whenever more than one set is in-progress. If you remove only the banner, you have not unlocked per-row rendering; you have only hidden the reason no set resolves.

High: The proposal claims the architecture "collapses to one writer per session," but its own flows require additional persisted state that is not specified. proposal.md says checkout is released on close, while proposal.md through proposal.md still expect "previous orchestrator," stalled-checkout recovery, and history UI. Then proposal.md through proposal.md acknowledges unresolved /think* and same-session multi-writer cases. As written, this is not yet a simplification of snapshot + precedence; it is an unscoped lease protocol plus history plus conflict UI. The proposal needs an explicit state model before it can fairly claim reduced complexity.

Medium: "Check in at close_session" is not a universal boundary in this repo, so the design currently breaks portability across supported workflows. proposal.md assumes a single automated release path, but the schema contract in session-state-schema.md and session-state-schema.md explicitly supports both Full-tier router-written flows and Lightweight hand-maintained flows. If checkout release depends on close_session, the proposal must say who performs release for Lightweight/manual sets and for abandoned sessions that never hit close-out. Otherwise the architecture introduces a new mandatory manual step, which works against the stated UI-minimalism goal.

Open Questions

proposal.md through proposal.md reads like "different orchestrator on resume" should warn by default. Is that intended only for suspended same-session resumes? If not, it clashes with the documented normal handoff model in ai-led-session-workflow.md, which says any orchestrator can pick up between sessions.

The operator's original objection seems mostly about naming and affordance clarity, but the proposal spends most of its weight on a deeper architecture rewrite. I would explicitly separate "rename/relegate confusing UI surfaces now" from "replace precedence with checkout later," because those are not coupled decisions in the current draft.

Overall, I would not approve the checkout/check-in architecture from this proposal as written. The strongest near-term move is to keep the current precedence model, rename or relegate the manual action so it is clearly a declaration rather than a runtime setter, and treat Q6 and any checkout design as a follow-on only after the proposal defines the actual state machine, release authority, and multi-in-progress resolver changes.
