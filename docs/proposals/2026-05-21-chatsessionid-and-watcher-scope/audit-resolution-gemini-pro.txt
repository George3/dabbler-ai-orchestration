D1 — MVVM-with-scope-discipline — VERIFIED
**Verdict:** The proposed scope discipline is a correct and necessary architectural simplification.
**Reasoning:** Forbidding inference-based state updates from indirect signals (like config files) and mandating writes through a canonical API (`start_session`) directly addresses the root cause of the observed stale-state bug. The two permitted watcher paths (`session-state.json`, `checkout-conflicts/`) represent direct, intentional state changes, eliminating a major source of race conditions and incorrectness. This discipline strengthens the "single source of truth" principle established in Set 032.

---

D2 — MVC-shaped agent-facing API on top of MVVM internals — VERIFIED
**Verdict:** The proposed decoupling of the agent-facing API (MVC-shaped) from the UI (MVVM) is a coherent and robust design.
**Reasoning:** This separation of concerns correctly simplifies the agent's role to a stateless command-line actor. Agents do not need to be aware of the UI's reactive nature. The concern about agents needing to check state before destructive actions is valid, but it is fully compatible with this model. An agent reading `session-state.json` before writing is a standard controller action (read-then-write), not a violation of the decoupling. The `per-boundary` check cadence proposed in Q2 correctly addresses this need.

**Reasoning:** The proposal correctly identifies the need for a per-orchestrator discovery process and a robust fallback (`new_chat_id` CLI). This is the correct engineering approach. However, providing a definitive list of env vars now would be speculative. The audit set must verify these authoritatively.
**Cited concern (if REJECTED or REFINED):** "TBD via audit"

Q2 — Cadence of chatSessionId checks — VERIFIED
**Verdict:** The recommended "per-boundary" cadence provides the best balance of safety and performance.
**Reasoning:** "Per-task" checking would introduce significant I/O overhead for negligible safety gains in most scenarios. "On-demand" is too lax and could allow an agent to perform several operations with a stale identity before detection. "Per-boundary" ensures that critical entry/exit points and high-impact operations are guarded. "Destructive ops" should be defined as any operation with external side-effects that are non-trivial to reverse, with "any tool call that writes to the file system" being an excellent and safe default.

---

Q3 — Takeover UX when chatSessionId mismatches — REFINED
**Verdict:** The default behavior should be context-dependent, prioritizing safety by blocking user action until the conflict is resolved.
**Reasoning:** A non-blocking toast prompt introduces a significant risk: the user might miss it, leaving the agent and UI in a desynchronized, "no check-out" state which could lead to confusion or lost work. The failure mode of a disruptive prompt is less severe than the failure mode of operating with an incorrect identity.
**Cited concern (if REJECTED or REFINED):** "Toast prompt (non-blocking)"
**Suggested refinement:** Default to a **modal prompt** in GUI environments to force an explicit user decision. Default to a **CLI prompt** in terminal environments. The non-blocking toast should not be the default for this class of coordination error.

Q4 — chatSessionId on close: clear or retain? — REFINED
**Verdict:** The `orchestrator` block in `session-state.json` should be cleared, but the `chatSessionId` should be persisted in the event ledger for a complete audit trail.
**Reasoning:** The proposal's recommendation to clear the `orchestrator` block is correct; `session-state.json` must only reflect the *current* state of an active session. However, the `chatSessionId` is valuable audit information. Simply discarding it is a lost opportunity. The event ledger is the correct place for historical data.
**Cited concern (if REJECTED or REFINED):** "Recommendation under audit: **clear on close** (parsimony...)"
**Suggested refinement:** Implement "clear on close" for `session-state.json`. In addition, augment the `closeout_succeeded` event schema in `session-events.jsonl` to include the `engine`, `provider`, and `chatSessionId` of the entity that performed the check-in.

Q5 — Migration tolerance for in-flight sets without chatSessionId — VERIFIED
**Verdict:** The proposed hybrid "tolerant-on-read, strict-on-write" approach is the correct and standard pattern for this type of migration.
**Reasoning:** This strategy provides a seamless user experience for legacy sets while guaranteeing that all subsequent state transitions adhere to the new, stricter rules. The concern about a race condition between two new chat instances is valid, but it is effectively mitigated by the prerequisite file-level locking on `session-state.json` for all write operations. The first writer will win the lock and populate the field; the second will be forced to re-read and will correctly fail the now-strict identity check.

---

Q6 — `requireExplicitTakeover` setting — VERIFIED
**Verdict:** The setting is a necessary escape hatch for legitimate power-user workflows and is not premature optimization.
**Reasoning:** The proposal correctly anticipates the "modal prompt fatigue" risk for users who intentionally run multiple, concurrent chat sessions against the same repository (e.g., for parallel refactoring and test generation). Forcing a takeover prompt in this known-safe, single-user scenario would be prohibitive friction. Providing a non-default option to disable the prompt respects this workflow while maintaining a safe-by-default posture for all other users.

---

Q7 — Watcher-scope policy as a code-level invariant — VERIFIED
**Verdict:** The recommended "convention test" is the most effective and pragmatic mechanism for enforcing the watcher discipline.
**Reasoning:** Documentation alone is insufficient as it is not automatically enforced. A custom lint rule adds tooling complexity. A convention test that greps the codebase for disallowed watcher patterns and maintains an explicit allowlist is simple to implement, runs with existing CI/CD, and serves as self-documenting code. When a legitimate exception arises, updating the test's allowlist is a straightforward, auditable change that requires explicit justification in the source code.
