1. Severity: Major
   Category: correctness
   Location: ai_router/path_aware_critique.py:read_path_aware_critique; ai_router/path_aware_critique.py:has_path_aware_critique_record; ai_router/close_session.py path-aware-critique gate branch
   Description: The durable policy can be silently downgraded to `none` by an unreadable or corrupt `activity-log.json`. `read_path_aware_critique()` catches `OSError`/`json.JSONDecodeError` and returns the default `none`, and `has_path_aware_critique_record()` catches the same failures and returns `False`. `close_session.py` gates on `pac_level != PATH_AWARE_CRITIQUE_NONE` before it ever calls `validate_path_aware_critique_gate()`, so a broken activity log skips the Set 066 gate entirely. I reproduced this locally with the shipped code: an invalid `activity-log.json` produced `read_path_aware_critique(...) == "none"`, `has_path_aware_critique_record(...) == False`, and `validate_path_aware_critique_gate(...)` returned `applicable=False, ok=True`. That violates the load-bearing claim that an already-recorded `required` policy cannot be silently disarmed. Concrete fix: represent "durable record unreadable" as a separate state/result and make the close path treat it as a gate problem or explicit warning, not as an implicit opt-out.

2. Severity: Major
   Category: contract-drift
   Location: ai_router/path_aware_critique.py:validate_path_aware_critique_artifact; ai_router/path_aware_critique.py:validate_path_aware_critique_gate; docs/path-aware-critique-schema.md:122-123
   Description: The runtime gate treats `sessionSetName` and the artifact's `pathAwareCritique` as decorative, even though the contract docs say they identify the covered set (the directory basename) and the policy level the artifact was produced under. The Python validator checks only that `sessionSetName` is non-empty and `pathAwareCritique` is in the enum; `validate_path_aware_critique_gate()` never compares either field to the active set name or the recorded policy level. I reproduced this locally with the shipped code: a set whose durable record was `required` still passed the gate with a `path-aware-critique.json` whose `sessionSetName` was `some-other-set` and whose `pathAwareCritique` was `advisory`. That means a copied or stale artifact from another set can satisfy a `required` gate for the wrong set. Concrete fix: in the gate path, compare the artifact's `sessionSetName` to `Path(session_set_dir).name` and its `pathAwareCritique` to the recorded level before accepting it.
