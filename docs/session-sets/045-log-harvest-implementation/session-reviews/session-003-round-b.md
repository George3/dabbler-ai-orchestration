- **Issue:** The `harvest` function allows a single native session to be bound by multiple launches, leading to duplicated event streams. The binding logic iterates through launches and considers the full, unmodified list of native sessions each time. A native session should be consumed after its first successful binding.
  - **Location:** `ai_router/joiner/schema.py`, `harvest()`
  - **Fix:** Prevent a native session from being bound more than once. After a launch successfully binds to a native session, that native session must be removed from the pool of candidates available to subsequent launches. The simplest way is to maintain a set of bound native IDs and add a check to the `candidates` list comprehension (e.g., `if (ns.engine, ns.conv_id) not in already_bound_ids`).

- **Issue:** Filtering in `harvest` can cause native sessions to be silently dropped from the output. Filters (`workspace_cwd`, `since`) are applied to a launch record *after* it has been used for binding. If the launch is filtered out, both the launch record and its bound native's events are suppressed. Because the native session was marked as bound, it is also suppressed from the free-running emission loop, causing it to disappear entirely, even if the native session itself would have matched the filter criteria.
  - **Location:** `ai_router/joiner/schema.py`, `harvest()`
  - **Fix:** Apply the `workspace_cwd` and `since` filters to a launch *before* attempting to find candidate native sessions for it. A launch that does not match the filter criteria should not be considered for binding at all.

- **Issue:** The `normalize_engine` function is brittle and only addresses known symptoms (`-code`, `-cli`) rather than the root cause of engine name variations. It would fail to match plausible variants like `"anthropic-claude"` vs `"claude"`.
  - **Location:** `ai_router/joiner/schema.py`, `normalize_engine()`
  - **Fix:** Implement a more robust canonicalization strategy. A simple approach is to check if the canonical name (e.g., "claude") is a substring of the candidate name (e.g., "anthropic-claude", "claude-code") after both are lowercased. A more robust long-term solution would be an alias map (e.g., `{"claude": ["claude-code", "anthropic-claude"]}`).
