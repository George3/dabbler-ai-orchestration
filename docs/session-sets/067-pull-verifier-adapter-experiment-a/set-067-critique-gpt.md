2. Severity: Major. Category: correctness. Location: `ai_router/pull_critique.py:255,260,294-295,451`; `ai_router/docs/pull-verifier.md:112,132,148`; `ai_router/CHANGELOG.md:60-64`. Description: the producer can emit a gate-passing artifact from the wrong review surface. The code says the run is over a "read-only repository sandbox" and claims it "refuses to write a gate-failing artifact," but the implementation defaults `sandbox_dir` to `Path.cwd()`, and the CLI advertises the same current-working-directory default. Nothing derives the sandbox from the repo that contains `session_set_dir`. So `python -m ai_router.pull_critique .` from inside the set directory, or from any other subdirectory, reviews only that subtree, not the repository that the docs/changelog describe. The artifact can still be written and pass the close-out gate because validation checks structure, provider distinctness, and identity fields, not whether the critic could actually see the repo. That is exactly a gate-passing-but-bogus artifact path. Concrete fix: derive the default sandbox from the git repo root containing `session_set_dir` instead of `Path.cwd()`, or refuse to write unless the sandbox is that repo root; if sandbox override remains, stamp and validate the reviewed root explicitly so the gate can reject underscoped artifacts.

3. Severity: Major. Category: correctness. Location: `ai_router/pull_verifier.py:1324-1358,1379-1380`; `ai_router/CHANGELOG.md:41-47`. Description: the S4 budget-aware force still does not guarantee a verdict before an empty budget stop. The guard reserves only the last call's measured size, but `last_call_tokens`/`last_call_cost` start at zero and ceilings are checked only at loop top. That leaves an immediate hole on the first oversized probe, and on any later probe that is much larger than the previous one: `force_verdict` is still false, the call overshoots the budget, and the next iteration exits `STOP_TOKEN_BUDGET` or `STOP_COST_CEILING` with no verdict. A local replay of the current code with `token_budget=100` and a single first probe that reports `80` input + `80` output tokens produced `{'ok': False, 'stop_reason': 'token-budget', 'api_turns': 1, 'force_flags': [False], 'tool_calls': 1}`. That contradicts the changelog's stronger claim that the new logic makes a verbose prober "commit a verdict instead of stopping empty." The tool contract is honest that token/cost caps are post-hoc; the bug is the stronger anti-empty-stop claim, not the post-hoc model itself. Concrete fix: reserve explicit first-turn headroom from configured worst-case call size, or after an overshooting probe allow one bounded forced-verdict turn before honoring the stop, then document the exact guarantee that remains.
